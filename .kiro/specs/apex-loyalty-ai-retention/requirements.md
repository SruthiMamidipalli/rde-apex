# Requirements Document

## Introduction

Apex Retail's loyalty platform investment of $2.1M has not reduced churn, which remains at 28% (3x industry benchmark). Six siloed systems hold churn signals but lack integration, and manual retention workflows take 4-7 days — too slow to intervene effectively. This feature delivers an AI-powered agentic retention system that cross-references all six data sources, computes composite churn scores, generates personalized retention offers, and enables human-in-the-loop approval — all within a 2-minute end-to-end cycle. The prototype targets Azure Sandbox deployment with mock data, a React dashboard, and MCP tool exposure for demo readiness.

## Glossary

- **Churn_Score_Engine**: The computation module that combines weighted signals from all six source systems into a single composite churn score (0-100) per customer
- **Retention_Agent**: The AI agent layer powered by Claude models via Amazon Bedrock that reasons across customer data, generates insights, retention briefs, and outreach content
- **Model_Router**: The intelligent routing service that directs AI tasks to the most cost-effective model (Sonnet for complex reasoning, Haiku for lightweight generation) while tracking token usage and estimated costs
- **Dashboard**: The React.js frontend providing visualizations of customer risk, signal breakdowns, and approval workflows
- **MCP_Server**: The Model Context Protocol server that exposes churn scoring, offer generation, and signal retrieval as callable tools
- **Approval_System**: The human-in-the-loop component enabling CRM analysts to approve or override AI-generated retention recommendations with a single click
- **Mock_Data_Layer**: The simulated data store representing all six source systems (Salesforce CRM, Shopify, Yotpo, Klaviyo, Zendesk, Google Analytics) using JSON or Cosmos DB
- **Retention_Brief**: A structured, cited document generated by the Retention_Agent containing customer risk analysis, signal breakdown, and recommended actions
- **Outreach_Content**: Multi-channel message drafts (email, SMS, push notification) personalized to a customer's churn signals and loyalty tier
- **Risk_Level**: A categorical classification of customer churn risk: LOW (0-25), MEDIUM (26-50), HIGH (51-75), CRITICAL (76-100)
- **Signal**: A data point from one of the six source systems indicating customer behavior change relevant to churn prediction
- **Customer_Tier**: The loyalty tier classification from Yotpo (e.g., Bronze, Silver, Gold, Platinum) used to match offer magnitude
- **Audit_Trail**: A timestamped log of every AI recommendation, data source consulted, confidence score, and human decision

## Requirements

### Requirement 1: Mock Data Layer

**User Story:** As a developer, I want a simulated data layer representing all six source systems, so that the prototype can demonstrate end-to-end functionality without real integrations.

#### Acceptance Criteria

1. THE Mock_Data_Layer SHALL provide customer records with engagement scores representing Salesforce CRM data
2. THE Mock_Data_Layer SHALL provide transaction data including average order value and discount usage representing Shopify data
3. THE Mock_Data_Layer SHALL provide loyalty points accrued and redemption history representing Yotpo data
4. THE Mock_Data_Layer SHALL provide email open rates, click rates, and SMS engagement metrics representing Klaviyo data
5. THE Mock_Data_Layer SHALL provide support ticket history with sentiment and resolution status representing Zendesk data
6. THE Mock_Data_Layer SHALL provide session frequency, browsing duration, and page-view patterns representing Google Analytics data
7. THE Mock_Data_Layer SHALL contain a minimum of 20 customer profiles with varied risk levels across all four Risk_Level categories
8. WHEN a customer record is requested by customer ID, THE Mock_Data_Layer SHALL return the consolidated data from all six simulated systems within 500ms

### Requirement 2: Composite Churn Score Calculation

**User Story:** As a CRM Analyst, I want a composite churn score calculated from all six data sources, so that I can identify at-risk customers with a single metric.

#### Acceptance Criteria

1. WHEN a customer ID is provided, THE Churn_Score_Engine SHALL compute a composite score between 0 and 100 using weighted signals from all six source systems
2. THE Churn_Score_Engine SHALL assign configurable weights to each source system signal (e.g., transaction recency 25%, engagement drop 20%, support sentiment 15%, session decline 15%, loyalty inactivity 15%, email disengagement 10%)
3. WHEN the composite score is computed, THE Churn_Score_Engine SHALL classify the customer into a Risk_Level: LOW (0-25), MEDIUM (26-50), HIGH (51-75), or CRITICAL (76-100)
4. WHEN the composite score is returned, THE Churn_Score_Engine SHALL include the individual signal contributions with source system attribution
5. THE Churn_Score_Engine SHALL recalculate scores when new signal data is available from the Mock_Data_Layer
6. WHEN a signal from any single source system is missing, THE Churn_Score_Engine SHALL compute the score using available signals and normalize the weights accordingly

### Requirement 3: AI Agent Orchestration

**User Story:** As a CRM Analyst, I want an AI agent that reasons across all data sources and generates actionable insights, so that I can understand why a customer is at risk and what to do about it.

#### Acceptance Criteria

1. WHEN a customer's composite score exceeds the configurable threshold (default: 50), THE Retention_Agent SHALL initiate an analysis workflow querying all six simulated systems
2. THE Retention_Agent SHALL use Azure OpenAI GPT-4 to reason across the combined customer data and identify the primary churn drivers
3. WHEN the Retention_Agent generates an insight, THE Retention_Agent SHALL cite the specific source system and data point supporting the insight
4. THE Retention_Agent SHALL complete the full analysis and recommendation cycle within 30 seconds of initiation
5. IF the Retention_Agent encounters an error from Bedrock, THEN THE Retention_Agent SHALL retry once and log the failure with a descriptive error message

### Requirement 4: Personalized Offer Generation

**User Story:** As a CRM Analyst, I want retention offers matched to the specific churn signal pattern and customer tier, so that interventions are relevant and effective.

#### Acceptance Criteria

1. WHEN the Retention_Agent identifies churn drivers, THE Retention_Agent SHALL generate a personalized retention offer matched to the dominant signal type
2. THE Retention_Agent SHALL scale offer value based on the customer's Customer_Tier (higher tier receives higher-value offers)
3. WHEN the churn signal is transaction-related (declining AOV or purchase frequency), THE Retention_Agent SHALL generate discount or bonus points offers
4. WHEN the churn signal is engagement-related (email/SMS disengagement), THE Retention_Agent SHALL generate re-engagement content offers (exclusive access, personalized recommendations)
5. WHEN the churn signal is support-related (negative sentiment, unresolved tickets), THE Retention_Agent SHALL generate service recovery offers (priority support, compensation)
6. THE Retention_Agent SHALL include a confidence score (0-100) with each generated offer indicating the predicted effectiveness

### Requirement 5: Retention Brief Generation

**User Story:** As a CRM Analyst, I want a structured retention brief per customer that summarizes risk, signals, and recommended actions with citations, so that I can make informed approval decisions quickly.

#### Acceptance Criteria

1. WHEN the Retention_Agent completes analysis for a customer, THE Retention_Agent SHALL generate a Retention_Brief containing: customer summary, risk classification, signal breakdown, recommended offer, and outreach strategy
2. THE Retention_Brief SHALL cite specific data points from source systems for each claim (e.g., "Transaction frequency dropped 60% over 30 days [Shopify]")
3. THE Retention_Brief SHALL include a comparison of current behavior versus the customer's historical baseline
4. THE Retention_Brief SHALL be formatted in structured sections with headers for easy scanning by CRM analysts
5. WHEN the Retention_Brief is generated, THE Retention_Agent SHALL complete generation within 20 seconds

### Requirement 6: Multi-Channel Outreach Content

**User Story:** As a CRM Analyst, I want AI-generated outreach drafts for email, SMS, and push notifications, so that I can launch retention campaigns immediately after approval.

#### Acceptance Criteria

1. WHEN a retention offer is generated, THE Retention_Agent SHALL produce outreach content drafts for three channels: email, SMS, and push notification
2. THE Outreach_Content for email SHALL include a subject line, body text with personalization tokens, and a call-to-action aligned with the offer
3. THE Outreach_Content for SMS SHALL be limited to 160 characters and include the core offer and a short link placeholder
4. THE Outreach_Content for push notification SHALL be limited to 50 characters for the title and 100 characters for the body
5. THE Outreach_Content SHALL match the tone appropriate to the customer's Risk_Level (urgent for CRITICAL, warm for HIGH, informational for MEDIUM)
6. THE Outreach_Content SHALL reference the customer by name and include the specific offer details

### Requirement 7: Human-in-the-Loop Approval Workflow

**User Story:** As a CRM Analyst, I want to approve or override AI-generated retention recommendations with a single click, so that I maintain control over customer communications.

#### Acceptance Criteria

1. WHEN the Retention_Agent generates a recommendation, THE Approval_System SHALL present the Retention_Brief, offer, and outreach drafts to the assigned CRM Analyst
2. THE Approval_System SHALL provide a single-click approve action that marks the recommendation as approved and logs the decision
3. THE Approval_System SHALL provide an override action allowing the CRM Analyst to modify the offer or outreach content before approval
4. WHEN a recommendation is approved, THE Approval_System SHALL log the approver identity, timestamp, and whether the original recommendation was modified
5. WHEN a recommendation for a CRITICAL Risk_Level customer involves an offer exceeding a configurable value threshold, THE Approval_System SHALL escalate to the DRI for approval
6. THE Approval_System SHALL display the time elapsed since signal detection alongside each pending recommendation

### Requirement 8: Customer Risk Dashboard

**User Story:** As a CRM Analyst, I want a visual dashboard showing all at-risk customers with their churn scores, risk levels, and signal breakdowns, so that I can prioritize my retention efforts.

#### Acceptance Criteria

1. THE Dashboard SHALL display a list of customers sorted by composite churn score in descending order
2. THE Dashboard SHALL display each customer as a card with color coding: green for LOW, amber for MEDIUM, red for HIGH, dark red for CRITICAL
3. THE Dashboard SHALL include a signal breakdown visualization (radar or spider chart) showing each source system's contribution to the churn score
4. WHEN a CRM Analyst selects a customer card, THE Dashboard SHALL display the full Retention_Brief, generated offer, and outreach drafts
5. THE Dashboard SHALL provide a timeline view showing signal detection timestamp, agent action timestamp, and current status
6. THE Dashboard SHALL support both dark mode and light mode themes
7. THE Dashboard SHALL display aggregate metrics: total at-risk customers, average churn score, campaigns pending approval, and campaigns launched

### Requirement 9: Before vs After Comparison

**User Story:** As an Executive, I want to see a side-by-side comparison of the manual retention process versus the AI agentic process, so that I can understand the ROI of the system.

#### Acceptance Criteria

1. THE Dashboard SHALL include a dedicated comparison panel showing manual process metrics alongside AI agentic process metrics
2. THE Dashboard SHALL display campaign launch time comparison: manual (4-7 days) versus agentic (target under 2 minutes)
3. THE Dashboard SHALL display churn detection timing comparison: reactive versus 14 days proactive
4. THE Dashboard SHALL display offer personalization rate comparison: manual (8%) versus agentic (85%+)
5. THE Dashboard SHALL display analyst hours per segment comparison: manual (4-8 hours) versus agentic (under 5 minutes)
6. THE Dashboard SHALL display projected annual churn rate comparison: current (28%) versus target (under 14%)

### Requirement 10: MCP Server Exposure

**User Story:** As a developer, I want core capabilities exposed as MCP tools, so that external systems and AI agents can invoke churn scoring and offer generation programmatically.

#### Acceptance Criteria

1. THE MCP_Server SHALL expose a `calculate_churn_score` tool that accepts a customer ID and returns the composite score with signal breakdown
2. THE MCP_Server SHALL expose a `generate_retention_offer` tool that accepts a customer profile and signals and returns a personalized offer with confidence score
3. THE MCP_Server SHALL expose a `get_customer_signals` tool that accepts a customer ID and returns all signals from the six simulated systems
4. THE MCP_Server SHALL expose a `generate_outreach_content` tool that accepts a customer profile, offer details, and channel type and returns formatted content
5. THE MCP_Server SHALL expose a `get_at_risk_customers` tool that accepts a score threshold and returns a list of customers exceeding that threshold
6. WHEN an MCP tool is invoked with invalid parameters, THE MCP_Server SHALL return a structured error response with a descriptive message
7. THE MCP_Server SHALL include input schema definitions for each exposed tool following the MCP specification

### Requirement 11: Audit Trail

**User Story:** As a Loyalty Ops Manager, I want every AI recommendation logged with sources, timestamps, and confidence scores, so that I can review decision history and ensure compliance.

#### Acceptance Criteria

1. WHEN the Retention_Agent generates a recommendation, THE Audit_Trail SHALL log the customer ID, timestamp, composite churn score, risk level, and source systems consulted
2. WHEN the Retention_Agent generates an offer, THE Audit_Trail SHALL log the offer type, value, confidence score, and matched churn signal
3. WHEN the Approval_System records a decision, THE Audit_Trail SHALL log the approver identity, decision (approve/override), timestamp, and any modifications made
4. THE Audit_Trail SHALL be queryable by customer ID, date range, risk level, and approver
5. THE Audit_Trail SHALL retain all entries for the duration of the prototype demonstration
6. WHEN the Dashboard displays a customer's history, THE Dashboard SHALL show the complete Audit_Trail for that customer in chronological order

### Requirement 12: Agentic Workflow Trigger

**User Story:** As a CRM Analyst, I want the system to automatically detect churn risk signals and initiate the retention workflow, so that at-risk customers are flagged proactively without manual monitoring.

#### Acceptance Criteria

1. WHEN a customer's composite churn score exceeds the configurable threshold (default: 50), THE Retention_Agent SHALL automatically initiate the retention workflow for that customer
2. WHEN a Google Analytics session frequency drops by more than 40% over a 14-day window, THE Retention_Agent SHALL flag the customer for immediate scoring
3. THE Retention_Agent SHALL process triggered workflows in priority order based on Risk_Level (CRITICAL first, then HIGH, MEDIUM, LOW)
4. WHEN the retention workflow completes, THE Retention_Agent SHALL deliver the Retention_Brief, offer, and outreach drafts to the Approval_System within 2 minutes of the initial trigger
5. THE Dashboard SHALL display the time elapsed from signal detection to recommendation delivery for each customer

### Requirement 13: Prototype Technical Constraints

**User Story:** As a developer, I want clear technical boundaries for the prototype, so that the team can deliver a demo-ready system within the 1-day build constraint.

#### Acceptance Criteria

1. THE Dashboard SHALL be implemented using React.js with a modern component library
2. THE Retention_Agent SHALL use Claude (Anthropic) models via Amazon Bedrock as the reasoning engine, with intelligent model routing based on task complexity
3. THE Mock_Data_Layer SHALL store data in JSON files or Azure Cosmos DB
4. THE MCP_Server SHALL be implemented using Python (FastAPI) or Node.js (Express)
5. THE Dashboard SHALL be deployable to Azure App Service or run locally for demonstration
6. THE Mock_Data_Layer SHALL be seeded with realistic data on application startup without requiring manual data entry

### Requirement 14: Intelligent Model Routing for Cost Optimization

**User Story:** As an Enterprise Architect, I want the system to route AI tasks to different LLM models based on task complexity, so that we demonstrate cost optimization without sacrificing quality for complex reasoning tasks.

#### Acceptance Criteria

1. THE Model_Router SHALL route complex reasoning tasks (churn analysis, offer generation, retention briefs) to Claude 3.5 Sonnet
2. THE Model_Router SHALL route lightweight tasks (outreach content drafting, summarization, formatting) to Claude 3 Haiku
3. THE Model_Router SHALL maintain a configurable task-to-model mapping that can be updated without code changes
4. WHEN any AI task is completed, THE Model_Router SHALL log the model used, input/output token counts, and estimated cost
5. THE Dashboard SHALL display a cost optimization panel showing: total cost, cost per model tier, all-Sonnet baseline cost, and percentage savings achieved
6. THE Model_Router SHALL achieve at least 60% cost reduction compared to routing all tasks to the most capable model
7. WHEN a model invocation fails, THE Model_Router SHALL fall back to the next available model tier and log the fallback event
